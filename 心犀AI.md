# 「心犀AI」—— 智能婚恋匹配系统项目企划书

## 一、 项目背景与痛点分析

### 1.1 行业背景
随着当代年轻人生活节奏的加快，社交圈子受限，线上婚恋交友需求持续增长。然而，传统的线上婚恋平台正在面临用户信任度下降、流失率高的困境。

### 1.2 核心痛点
*   **痛点一：硬性匹配死板。** 传统平台主要依靠传统的 SQL 条件筛选（如身高、收入、地区），导致匹配结果过于机械，忽略了“三观、性格、生活细节”等深层次的软性契合度。
*   **痛点二：语义搜索缺失。** 用户无法使用自然语言表达自己真实复杂的择偶期望，系统无法理解类似“希望对方性格温和，周末喜欢安静，对猫咪过敏但喜欢小动物”这种复杂的复合语义。
*   **痛点三：匹配推荐缺乏解释。** 传统平台通常直接展示候选人列表，用户不明白“系统为什么推荐这个人”，缺乏说服力和人情味，难以建立信任。

---

## 二、 产品定位与核心价值

本系统旨在打造一个**懂用户、有温度、高效率**的 AI 辅助婚恋平台。其核心竞争优势在于：

1.  **意图深度解析**：引入大模型作为智能体（Agent）的大脑，深度拆解用户的个人特征和潜在的择偶需求。
2.  **硬性指标+软性契合双路检索**：采用混合检索技术，用 SQL/Metadata 确保硬性指标（性别、地域、年龄）“一票否决”的绝对准确；用向量检索实现三观、性格和兴趣爱好的“模糊契合”。
3.  **自适应循环推荐**：若初始检索结果不佳，Agent 会自主进行反思与参数微调，拓宽或优化检索策略。
4.  **可视化匹配理由生成**：由 LLM 梳理双方契合点，自动生成富有温度的推荐信。

## 三、 技术架构方案

系统采用 **“Agent（智能体）+ 混合 RAG”** 架构，技术栈建议选用主流的开源生态。

```
                       ┌─────────────────────────┐
                       │    用户输入/修改个人资料    │
                       └────────────┬────────────┘
                                    ▼
                       ┌─────────────────────────┐
                       │  Agent大脑 (LLM 预分析)  │
                       └────────────┬────────────┘
                                    │
            ┌───────────────────────┴───────────────────────┐
            ▼ (若修改结构化数据)                             ▼ (若修改介绍文本)
  ┌───────────────────┐                           ┌───────────────────┐
  │ 直接更新元数据字段 │                           │ 调用 Embedding 接口│
  └─────────┬─────────┘                           └─────────┬─────────┘
            │                                               ▼
            │                                     ┌───────────────────┐
            │                                     │  UPSERT 覆盖新向量 │
            │                                     └─────────┬─────────┘
            └───────────────────────┬───────────────────────┘
                                    ▼
                     ┌─────────────────────────────┐
                     │ 混合检索 (元数据过滤+向量搜索) │
                     └──────────────┬──────────────┘
                                    ▼
                     ┌─────────────────────────────┐
                     │   Agent 后分析与匹配打分     │
                     └──────────────┬──────────────┘
                                    │
                     ┌──────────────┴──────────────┐
                     ├───► [满足条件] ───► 生成匹配推荐信
                     │
                     └───► [不满足条件] ──► Agent 自主调整策略，重新检索
```

### 3.1 核心组件
*   **推理大脑（LLM）**：建议采用中等参数量但指令遵循能力强的模型（如 GPT-4o-mini 或 Qwen-Max），用于分析意图、打分及生成推荐语。
*   **向量化模型（Embedding）**：用于将用户的自由文本转化为数学高维向量。
*   **混合型向量数据库**：如 PostgreSQL (配 pgvector 插件) 或 Milvus。用于支持元数据（Metadata）的高效 SQL 过滤，以及向量（Vector）的相似度检索 [1, 2]。

### 3.2 数据结构设计 (User Profile Schema)
每个用户的数据在库中以混合模式存储 [1]：
*   **元数据字段（用于硬性剔除）**：`user_id`, `gender` (性别), `target_gender` (期望性别), `age` (年龄), `city` (城市) 等。
*   **向量字段（用于软性契合）**：`profile_vector` （由用户的“性格描述”、“择偶期望文本”综合转换而成的 1536 维向量） [1]。

## 四、 核心业务工作流

系统运行分为两个核心模块：**用户更新流** 与 **匹配检索流**。

### 4.1 用户信息更新流（实时性保障）
为了保证用户修改信息后能被即时搜索到，系统使用以下分流更新机制：
1.  **极速通道（属性修改）**：若用户仅修改年龄、城市或婚姻状况等，直接将新数据更新至向量库的元数据（Metadata）中 [8]。**耗时 < 10ms，无需重新计算向量**。
2.  **AI 通道（文本修改）**：若用户修改了“关于我”或“期望的Ta”等文本，系统触发事件，重新调用 Embedding 接口生成 1536 维新向量，并通过 `UPSERT` 写入数据库，替换原有向量 [6, 7]。**整体耗时 < 500ms，对用户近乎实时生效**。

### 4.2 匹配检索与推荐流（Agentic RAG 闭环）
当用户点击“寻找缘分”时，触发以下四步闭环：

#### 第一步：LLM 意图解析（Query Rewriting）
Agent 读取当前用户的资料与期待，将其拆解并重构：
*   **提取硬性过滤条件**：例如 `gender = 'female' AND age BETWEEN 25 AND 30`。
*   **重写检索向量文本**：将感性的叙述重写为适合搜索的客观词簇（如：“宅、喜安静” 转化为 “喜欢阅读、看电影、室内活动、性格内敛” 的特征向量）。

#### 第二步：混合检索（Pre-filtering + Semantic Search）
向向量数据库发起检索：
*   数据库**先执行结构化过滤**，将不符合硬性条件的人100%剔除，缩小候选集 [3, 4]。
*   在剩下的人群中，计算其余特征向量的相似度，**返回排名前 10 的候选人**。

#### 第三步：LLM 后分析与精排（Reasoning）
Agent 接收到 10 位候选人的完整文本资料，进行深度交叉对比，计算综合契合指数。

#### 第四步：反馈循环与输出（Agentic Loop）
*   **若满足阈值**：Agent 挑选契合度最高的前 3 名，并利用 LLM 分别生成定制化的“匹配信”，说明双方在三观或习惯上的绝妙重合点。
*   **若不满足阈值**：Agent 启动反思机制。“由于‘同城’条件限制，无法匹配到相似爱好的对象。自动将地理范围扩大至‘省份同城’，重新返回第一步进行检索。”（此过程由程序后台自动循环，限制最大循环次数为3次，以保证接口响应速度）。

## 五、 商业模式与应用价值

### 5.1 提升用户留存与付费转化
*   **有温度的推荐**：告别冷冰冰的数据表，用户看到的是由 AI 撰写的专属“红娘推荐语”，极大增强了产品的亲和力和用户的好奇心。
*   **会员增值服务**：
    *   基础版会员：仅支持硬性指标筛选（SQL 模式）。
    *   高级/尊享会员：解锁“心犀AI智能体匹配”（Agentic RAG），可无限次进行深度三观匹配。

### 5.2 降低人工红娘成本
通过 AI Agent 模拟了专业红娘的“分析-筛选-核对-介绍”全套流程，可作为线上轻量化红娘服务，极大地降低了平台雇佣人工客服或红娘的运营成本。

## 七、 风险控制与应对策略

1.  **数据隐私与安全（关键）**：
    *   *风险*：婚恋数据涉及用户大量私密信息。
    *   *应对*：敏感隐私数据（如真实姓名、联系方式、精确资产）不参与 Embedding 计算，仅保留在安全的关系型数据库中。向量数据库中只保存用于匹配的抽象描述向量 [1]。
2.  **大模型幻觉与偏见**：
    *   *风险*：大模型在生成匹配理由时可能会编造用户不存在的爱好。
    *   *应对*：在 Prompt 设计中，加入严格的约束规范（System Prompt），限制大模型只能基于提供的检索结果（Context）进行润色和总结，禁止进行无事实基础的推导。
3.  **大模型接口调用成本**：
    *   *风险*：高频调用大模型可能导致运营成本高昂。
    *   *应对*：实施分类更新（元数据修改不调大模型）；对多次未匹配成功的用户，在前端设置合理的冷却时间（Cool-down），限制短时间内高频重复检索。
